代码可读性与代码优化

Written by 周啟尧 with ♥ on January 19, 2020 in 其他

优化原则

工作中，尤其是在一个团队里面我们的代码常常是要写给别人看的，因此如果不是遇到性能瓶颈，我们应该优先考虑代码的易读性。
而无论我们如何对代码作出优化，逻辑的复杂度始终是不会改变的，如果实现需求本身的逻辑是复杂的，那么代码看起来就应该是复杂的，如果逻辑是简单的，那么代码看起来也应该是简单的。

变量命名

词性

  • 普通的变量/属性：名词

    // 不要使用a b c等没有意义的变量名
let person = {
  name:'jack',
  age:'18'
}

  • 布尔类型的变量/属性：形容词/be动词/情态动词/{hasX}

    // 形容词
dead:false
// be动词
canListen:true
// 情态动词 - can
isTeacher:true
// hasXxx
hasChildren:true

  • 普通函数/方法：动词

    run()
drinkTea()
eat(fruit){}

  • 回调，钩子函数：用介词开头/动词的现在完成时

    // Vue就是一个很好的例子
created(){}
beforeMount(){}
mounted(){}
beforeUpdate(){}
updated(){}
...

  • 巧用前缀

    // DOM对象 加 el 前缀
// jQuery对象 加 $ 前缀
elDiv1.classList.add('active')
$div2.addClass('active')

一致性

  • 介词一致性
    如果使用了before + after 组合，就不要使用before + 动词现在完成时组合
  • 顺序一致性
    比如 updateFatherMoney 和 updateMoneyOfFather
  • 表里一致性
    函数名应该准确体现函数的功能，不能多也不能少。
  • 时间一致性
    随着代码的不断升级维护，当一个变量的含义不同于它一开始定义的含义时，应该及时更改。

模块化

我们说的MVC是宏观层面的模块化，放眼具体的代码，我们可以：
将具体实现某些功能的代码片段封装成函数；
如果函数与函数之间具有联系，可以进一步封装成对象。
拒绝面条式的代码。

模块化代码可以让代码更易读，比如我们可以通过看函数名就知道函数内部的代码大概实现了什么样的功能；此外，在之后的维护、代码升级或者需求更改时，我们也可以很快定位具体代码的位置。

常用技巧

  • 表驱动编程
  • 自说明代码 把别人关心的东西写在前面，尤其是在我们面向接口编程的时候。比如当别人在调用我们封装的函数时，别人要考虑传递的参数有哪些？我们就可以写在前面。

写在最后

优秀的程序员应该做到：只要代码经过你的手，都要比之前要好一点。
